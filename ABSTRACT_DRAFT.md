## บทคัดย่อ

ปัจจุบันผู้บริโภคให้ความสำคัญกับความปลอดภัยของส่วนผสมในผลิตภัณฑ์เครื่องสำอางมากขึ้น เนื่องจากสารเคมีบางชนิดอาจก่อให้เกิดการระคายเคืองหรืออาการแพ้ต่อผิวหนัง อย่างไรก็ตาม รายชื่อส่วนผสมบนฉลากผลิตภัณฑ์มักพิมพ์ด้วยตัวอักษรขนาดเล็กและประกอบด้วยชื่อสารเคมีที่มีความซับซ้อน ทำให้ผู้ใช้ทั่วไปตรวจสอบข้อมูลได้ยาก โครงงานนี้จึงพัฒนาระบบต้นแบบสำหรับดึงข้อมูลส่วนผสมของเครื่องสำอางจากภาพฉลากผลิตภัณฑ์ด้วยเทคนิค OCR โดยใช้ PaddleOCR (PP-OCRv6_medium) เป็นเครื่องมือรู้จำข้อความหลัก

งานวิจัยนี้ดำเนินการในลักษณะการวินิจฉัยและปรับแต่ง pipeline OCR เชิงประจักษ์แบบวนซ้ำ (iterative empirical pipeline engineering) แทนการเปรียบเทียบเทคนิค image preprocessing แบบดั้งเดิมจำนวนมาก โดยพบว่าการตั้งค่าฟีเจอร์ในตัวของโมเดลอย่างเหมาะสม (เปิดใช้ document unwarping และ textline orientation classification ซึ่งเดิมถูกปิดไว้) ร่วมกับการเพิ่มขอบภาพขาว (white border padding) 25% รอบภาพก่อนเข้าสู่ขั้นตอน OCR การเพิ่มคำศัพท์ที่ขาดหายไปในฐานข้อมูล INCI (curated vocabulary patch) และการปรับลด threshold การยอมรับกล่องข้อความของโมดูลตรวจจับข้อความ (text detection box threshold) ให้ผลดีขึ้นอย่างมีนัยสำคัญ โดยไม่ต้องอาศัยเทคนิค preprocessing เชิงภาพแบบดั้งเดิมเลย ใช้เทคนิค Fuzzy String Matching ด้วยไลบรารี RapidFuzz เทียบกับฐานข้อมูลชื่อสารเครื่องสำอาง (INCI) เพื่อแก้ไขคำที่ OCR อ่านผิดพลาดเป็นขั้นตอนสุดท้าย

ผลการทดลองบนภาพฉลากเครื่องสำอางจริงจำนวน 216 ภาพ พบว่าการแก้ไขการตั้งค่า pipeline หลัก (mkldnn, engine settings, ฐานข้อมูลคำศัพท์, การแก้ไขข้อความหลัง OCR) ทำให้ค่า Set F1 (ตัวชี้วัดระดับชื่อสาร) ดีขึ้นจาก 0.39 เป็น 0.55 (+41%) และการปรับลด threshold การตรวจจับข้อความเพิ่มเติม (จาก 0.45 เป็น 0.30) ทำให้ Set F1 ดีขึ้นอีกเป็น 0.57 พร้อมค่า Character Error Rate (CER) ลดลงจาก 0.32 เหลือ 0.26 คอขวดหลักที่เหลืออยู่คือค่า Recall ของโมดูลตรวจจับข้อความ ซึ่งประมาณ 62.7% ของชื่อสารที่พลาดเกิดจากโมดูลตรวจจับมองไม่เห็นข้อความเลย ไม่ใช่ปัญหาการแก้ไขข้อความหลัง OCR อีกต่อไป ชี้ให้เห็นว่าการพัฒนาต่อไปควรมุ่งไปที่คุณภาพ/ความละเอียดของภาพต้นทาง หรือการปรับแต่งโมเดลตรวจจับข้อความเฉพาะทาง (fine-tuning) มากกว่าการปรับค่าพารามิเตอร์ต่อไป

**คำสำคัญ:** OCR, PaddleOCR, Text Detection Threshold Tuning, Fuzzy Matching, RapidFuzz, Set F1-score, Character Error Rate

## Abstract (English)

Consumers are increasingly concerned about the safety of cosmetic ingredients, as certain chemicals may cause skin irritation or allergic reactions. However, ingredient lists on product labels are often printed in small fonts with complex chemical names, making them difficult for general users to verify. This project develops a prototype OCR system for extracting cosmetic ingredient information from product label images, using PaddleOCR (PP-OCRv6_medium) as the primary recognition engine.

Rather than comparing a large battery of classical image-preprocessing techniques, this work follows an iterative, empirical OCR-pipeline diagnosis-and-repair approach. The engine's built-in features were found to be misconfigured in an earlier iteration (document unwarping and textline-orientation classification had been disabled); re-enabling them, combined with a 25% white border padding added around each image before OCR, a curated vocabulary patch filling gaps in the reference INCI ingredient database, and lowering the text-detection module's box-acceptance threshold, produced substantial accuracy gains without any classical image-preprocessing step. RapidFuzz-based fuzzy string matching against the INCI database was retained as a post-OCR correction step.

On 216 real cosmetic label images, fixing the core pipeline configuration (mkldnn crash workaround, correct engine settings, vocabulary gaps, post-processing bugs) raised the ingredient-level Set F1-score from 0.39 to 0.55 (+41%). A subsequent detection-threshold adjustment (box_thresh 0.45 → 0.30) raised Set F1 further to 0.57 and reduced Character Error Rate (CER) from 0.32 to 0.26. The main remaining bottleneck is detector recall: roughly 62.7% of missed ingredient tokens are caused by the text detector failing to locate the text region at all, rather than by post-processing errors — indicating that further gains likely require better source-image resolution/quality or a fine-tuned, domain-specific text detector rather than further parameter tuning.

**Keywords:** OCR, PaddleOCR, Text Detection Threshold Tuning, Fuzzy Matching, RapidFuzz, Set F1-score, Character Error Rate

## 1.1 ความเป็นมาและความสำคัญของปัญหา

ในปัจจุบันผลิตภัณฑ์เครื่องสำอางและผลิตภัณฑ์ดูแลผิวได้รับความนิยมอย่างแพร่หลาย ผู้บริโภคจำนวนมากให้ความสำคัญกับการตรวจสอบส่วนผสมของผลิตภัณฑ์ก่อนการใช้งาน เนื่องจากสารบางชนิดอาจก่อให้เกิดการระคายเคือง อาการแพ้ หรือผลกระทบต่อสุขภาพผิว โดยเฉพาะผู้ที่มีผิวแพ้ง่าย อย่างไรก็ตาม รายชื่อส่วนผสมบนฉลากเครื่องสำอางมักแสดงด้วยตัวอักษรขนาดเล็ก มีความหนาแน่นสูง และประกอบด้วยชื่อสารเคมีที่มีความซับซ้อน ส่งผลให้ผู้ใช้ทั่วไปไม่สามารถอ่านและตรวจสอบข้อมูลได้อย่างสะดวก

เทคโนโลยี Optical Character Recognition (OCR) เป็นเทคโนโลยีที่ช่วยให้คอมพิวเตอร์สามารถรู้จำข้อความจากภาพได้โดยอัตโนมัติ ในปัจจุบัน เทคโนโลยี OCR ที่ใช้เทคนิค Deep Learning เช่น PaddleOCR สามารถรองรับข้อความที่มีรูปแบบหลากหลายและมีความซับซ้อนได้ดียิ่งขึ้น อย่างไรก็ตาม ในทางปฏิบัติ ภาพฉลากที่ผู้ใช้ถ่ายเองมักมีคุณภาพต่ำ เช่น ถ่ายในที่แสงน้อย ภาพสั่นหรือเบลอ และมุมกล้องเอียง ซึ่งส่งผลโดยตรงต่อความแม่นยำของ OCR นอกจากนี้ยังพบว่าโมดูล OCR สำเร็จรูปอย่าง PaddleOCR มีพารามิเตอร์การตั้งค่าจำนวนมากที่ส่งผลต่อความแม่นยำอย่างมีนัยสำคัญ (เช่น การเปิด/ปิดฟีเจอร์แก้ไขมุมมองเอกสาร หรือ threshold ของโมดูลตรวจจับข้อความ) ซึ่งหากตั้งค่าไม่เหมาะสมจะทำให้ระบบทำงานได้ต่ำกว่าศักยภาพที่แท้จริงของโมเดลมาก

โครงงานนี้จึงพัฒนาระบบต้นแบบสำหรับดึงข้อมูลส่วนผสมของเครื่องสำอางจากภาพฉลากผลิตภัณฑ์ด้วยเทคนิค OCR โดยมีเป้าหมายเพื่อนำไปพัฒนาต่อยอดเป็น**ระบบแนะนำสกินแคร์ที่เหมาะสมกับผู้ใช้แต่ละราย**ในอนาคต งานวิจัยนี้เน้นกระบวนการวินิจฉัยและแก้ไขการตั้งค่า pipeline OCR อย่างเป็นระบบผ่านการทดลองแบบมีการควบคุม (controlled ablation) ร่วมกับการเพิ่มขั้นตอนเสริมที่ออกแบบเฉพาะสำหรับลักษณะข้อมูล (ภาพแถบข้อความแคบ ความละเอียดต่ำ) แทนการอาศัยเทคนิคปรับปรุงคุณภาพภาพเชิงคลาสสิกจำนวนมาก เพื่อให้ได้ pipeline OCR ที่แม่นยำที่สุดสำหรับนำไปพัฒนาต่อและผนวกเข้ากับระบบแนะนำสกินแคร์ในขั้นต่อไป

## 1.2 วัตถุประสงค์ของโครงงาน

1. เพื่อพัฒนาระบบ OCR สำหรับดึงข้อความส่วนผสมจากภาพฉลากเครื่องสำอาง
2. เพื่อวินิจฉัยและแก้ไขการตั้งค่า pipeline ของ PaddleOCR (engine configuration, การตรวจจับข้อความ) ที่ส่งผลต่อความแม่นยำของการรู้จำข้อความจากภาพฉลากผลิตภัณฑ์
3. เพื่อพัฒนากระบวนการตรวจจับชื่อสารด้วยเทคนิค Fuzzy String Matching ร่วมกับฐานข้อมูล INCI ที่ผ่านการเสริมคำศัพท์ที่ขาดหายไป
4. เพื่อประเมินประสิทธิภาพของระบบด้วยตัวชี้วัดระดับชื่อสาร (Set Precision/Recall/F1-score, Ingredient Error Rate) ร่วมกับ Character Error Rate (CER) และ Word Error Rate (WER)

---

## 1.3 ขอบเขตของโครงงาน

1. ใช้ภาพฉลากผลิตภัณฑ์เครื่องสำอางภาษาอังกฤษเป็นหลัก
2. ใช้ **PaddleOCR 3.7.0 (PP-OCRv6_medium)** เป็นเครื่องมือหลักสำหรับการดึงข้อความจากภาพ รันบน CPU
3. วินิจฉัยและแก้ไขการตั้งค่า pipeline หลัก 4 ส่วน ได้แก่ (1) การตั้งค่าฟีเจอร์ในตัวของโมเดล (document unwarping, textline orientation, การปิด mkldnn เพื่อหลีกเลี่ยงบัค crash บนสภาพแวดล้อมที่ใช้) (2) การเพิ่มขอบภาพขาว (border padding) ก่อนเข้าสู่ OCR (3) การเสริมฐานข้อมูลคำศัพท์ INCI ที่ขาดหายไป และ (4) การปรับ threshold การตรวจจับข้อความ (text detection box threshold)
4. ใช้เทคนิค Fuzzy Matching (ไลบรารี RapidFuzz, `fuzz.ratio`, score cutoff 90) สำหรับตรวจจับชื่อสารจากฐานข้อมูลเครื่องสำอาง (INCI) หลัก 11,091 รายการ เสริมด้วยคำศัพท์ที่คัดสรรเพิ่มเติม 199 รายการ (รวม 11,290 รายการไม่ซ้ำ)
5. ใช้ชุดข้อมูลภาพฉลากเครื่องสำอางจริงจำนวน **216 ภาพ** และภาพจำลองสภาพแวดล้อม (data augmentation ด้วยไลบรารี albumentations) เพิ่มเติม 4 สภาพแวดล้อม (ภาพเบลอ, แสงน้อย, แสงกลางคืน, แสงในอาคาร) รวมเป็น 5 สภาพแวดล้อม **1,080 ภาพ** (การ validate เต็มรูปแบบของ 4 สภาพแวดล้อมเสริมยังอยู่ระหว่างดำเนินการ ดูหัวข้อ 4.5)
6. ประเมินผลด้วยตัวชี้วัดระดับชื่อสาร (Set Precision/Recall/F1-score, Ingredient Error Rate) เป็นหลัก ร่วมกับ CER/WER เป็นตัวชี้วัดเสริมระดับตัวอักษร/คำ

## 1.4 ประโยชน์ที่คาดว่าจะได้รับ

1. ได้โมดูล OCR ที่ผ่านการวินิจฉัยและแก้ไขการตั้งค่าอย่างเป็นระบบ พร้อมนำไปพัฒนาต่อยอดและผนวกเข้ากับ**ระบบแนะนำสกินแคร์ที่เหมาะสมกับผู้ใช้**ได้จริง
2. สามารถนำแนวทางการวินิจฉัยและปรับแต่ง pipeline OCR (engine configuration, detection threshold tuning) ไปประยุกต์ใช้กับงานวิเคราะห์ข้อความจากภาพในโมดูลอื่นของระบบแนะนำสกินแคร์ หรือระบบที่เกี่ยวข้องได้
3. เป็นแนวทางและหลักฐานเชิงประจักษ์ประกอบการตัดสินใจออกแบบ pipeline การอ่านฉลากของระบบแนะนำสกินแคร์ในขั้นพัฒนาต่อไป
4. เป็นประโยชน์ต่อการศึกษาและวิจัยด้าน Computer Vision และ Image Processing โดยเฉพาะในประเด็นที่ว่า**การตั้งค่าฟีเจอร์ในตัวของโมเดล OCR สำเร็จรูปอย่างเหมาะสม (engine configuration) อาจให้ผลกระทบต่อความแม่นยำมากกว่าการเพิ่มเทคนิค preprocessing เชิงภาพจำนวนมาก** และควรมีการทดสอบเชิงประจักษ์เพื่อวินิจฉัยจุดที่แท้จริงก่อนลงทุนพัฒนาเทคนิคเสริมที่ซับซ้อน

# บทที่ 2 ทฤษฎีและงานวิจัยที่เกี่ยวข้อง (ฉบับปรับปรุงครั้งที่ 2)

> ตัดหัวข้อทฤษฎีของเทคนิค preprocessing แบบดั้งเดิมที่ไม่ได้ใช้งานจริงในเวอร์ชันปัจจุบันออกทั้งหมด (Deskewing, CLAHE/Histogram Equalization, EDSR Super-Resolution, Tiled OCR, Multi-Tier Fallback, Greedy/Exhaustive Search) แทนที่ด้วยทฤษฎีของสิ่งที่ใช้งานจริง ได้แก่ การตั้งค่าฟีเจอร์ในตัวของ PaddleOCR, Border Padding, และหลักการของ Text Detection Post-processing (DB algorithm) ที่ปรับ threshold

## 2.1 Optical Character Recognition (OCR)

Optical Character Recognition (OCR) เป็นเทคโนโลยีที่ใช้สำหรับแปลงข้อความที่อยู่ในรูปแบบภาพให้อยู่ในรูปแบบข้อมูลดิจิทัลที่สามารถนำไปประมวลผลด้วยคอมพิวเตอร์ได้ โดยกระบวนการ OCR ประกอบด้วยหลายขั้นตอน ได้แก่ การตรวจจับตำแหน่งข้อความ (Text Detection) และการรู้จำตัวอักษร (Text Recognition)

ปัจจุบัน OCR ถูกนำไปประยุกต์ใช้อย่างแพร่หลายในงานด้านการแปลงเอกสารเป็นข้อมูลดิจิทัล การอ่านป้ายทะเบียนรถ การสแกนใบเสร็จ การรู้จำข้อความบนป้ายสินค้า และการวิเคราะห์ข้อมูลจากฉลากผลิตภัณฑ์ เทคโนโลยี OCR สมัยใหม่มีการประยุกต์ใช้เทคนิค Deep Learning เพื่อเพิ่มความสามารถในการรู้จำข้อความที่มีลักษณะซับซ้อน เช่น ตัวอักษรขนาดเล็ก ตัวอักษรเอียง หรือข้อความที่อยู่บนพื้นหลังหลากหลายรูปแบบ

ในโครงงานนี้ OCR ถูกนำมาใช้เพื่อดึงข้อมูลรายชื่อส่วนผสมของเครื่องสำอางจากภาพฉลากผลิตภัณฑ์ เพื่อพัฒนาเป็นโมดูลสำหรับผนวกเข้ากับระบบแนะนำสกินแคร์ที่เหมาะสมกับผู้ใช้

## 2.2 PaddleOCR

PaddleOCR เป็นไลบรารี OCR แบบ Open-source ที่พัฒนาโดย PaddlePaddle ซึ่งเป็นเฟรมเวิร์กด้าน Deep Learning ที่ได้รับการพัฒนาโดย Baidu โดย PaddleOCR ได้รับการออกแบบให้รองรับทั้งขั้นตอนการตรวจจับข้อความ (Text Detection) และการรู้จำข้อความ (Text Recognition) ภายในระบบเดียว รวมถึงมีโมดูลเสริมที่ทำงานก่อนขั้นตอนหลัก เช่น document orientation classification, document unwarping (แก้ไขความโค้ง/มุมมองของภาพเอกสาร) และ textline orientation classification (ตรวจจับทิศทางของแต่ละบรรทัดข้อความ)

โมดูลตรวจจับข้อความของ PaddleOCR (PP-OCRv6) ใช้สถาปัตยกรรม **DB (Differentiable Binarization)** ซึ่งทำนาย probability map ของตำแหน่งข้อความ แล้วแปลงเป็นกรอบข้อความ (bounding box/polygon) ด้วยขั้นตอน post-processing ที่มีพารามิเตอร์ควบคุมได้ (รายละเอียดในหัวข้อ 2.4) — จุดนี้สำคัญเพราะเป็นส่วนที่โครงงานนี้ปรับแต่งเพื่อเพิ่มความแม่นยำ

ในโครงงานนี้ใช้ PaddleOCR เวอร์ชัน **3.7.0** โมเดล **PP-OCRv6_medium** เป็น OCR Engine เพียงตัวเดียวสำหรับการดึงข้อความจากภาพฉลากเครื่องสำอาง รันบน **CPU** (สภาพแวดล้อมที่ใช้พัฒนามีข้อจำกัดด้าน GPU — ดูหัวข้อ 3.11) โดย**เปิดใช้งาน** document unwarping และ textline orientation classification (ต่างจากแนวทางเดิมที่ปิดฟีเจอร์เหล่านี้เพื่อทดสอบ preprocessing แบบแยกส่วน) เนื่องจากพบเชิงประจักษ์ว่าการเปิดใช้ทั้งสองฟีเจอร์นี้ช่วยลดค่า Character Error Rate ได้มากกว่าเทคนิค preprocessing เชิงคลาสสิกใด ๆ ที่เคยทดลอง

## 2.3 Border Padding (การเพิ่มขอบภาพก่อน OCR)

ภาพฉลากในชุดข้อมูลของโครงงานนี้ถูก crop มาเป็นแถบข้อความแคบ (median อัตราส่วนกว้าง:สูง ≈ 3:1) ซึ่งมักมีตัวอักษรอยู่ชิดขอบภาพมาก โมดูลตรวจจับข้อความของโมเดล OCR หลายตัว (รวมถึง PP-OCRv6) มักมองข้ามข้อความที่อยู่ชิดขอบภาพ เนื่องจากขั้นตอน post-processing ของโมเดลอาจตัดกรอบข้อความที่อยู่ติดขอบภาพทิ้ง หรือ resize ภาพเข้าสู่โมเดลด้วยอัตราส่วนที่ทำให้บริเวณขอบเสียรายละเอียด

Border Padding คือการเพิ่มขอบภาพสีขาวรอบภาพก่อนส่งเข้าสู่โมดูล OCR เพื่อ "ดัน" ข้อความที่เคยอยู่ชิดขอบให้ห่างจากขอบภาพจริงมากขึ้น ช่วยให้โมดูลตรวจจับข้อความมีโอกาสพบกรอบข้อความบริเวณนั้นมากขึ้น โดยไม่ต้องปรับแก้โมเดลหรือ pipeline ภายใน

## 2.4 การตรวจจับข้อความแบบ DB (Differentiable Binarization) และ Threshold การ Post-processing

โมดูลตรวจจับข้อความที่ใช้อัลกอริทึม DB จะทำนาย probability map แล้วแปลงเป็นกรอบข้อความผ่านขั้นตอน post-processing ที่มีพารามิเตอร์สำคัญ ได้แก่

- **thresh** — ค่า threshold ระดับ pixel ของ probability map เพื่อสร้าง binary map เบื้องต้น
- **box_thresh** — เกณฑ์คะแนนความเชื่อมั่นเฉลี่ยภายในกรอบข้อความที่ตรวจพบ กรอบใดมีคะแนนต่ำกว่าเกณฑ์นี้จะถูกทิ้งไป (ไม่นับเป็นข้อความที่ตรวจพบ) — ค่าเริ่มต้นของ PP-OCRv6_medium คือ 0.45
- **unclip_ratio** — อัตราส่วนขยายกรอบข้อความออกจากขอบตัวอักษรจริง เพื่อชดเชยกรอบที่ตรวจพบแคบเกินไปจากขั้นตอน binarization
- **limit_side_len** — ขนาดด้านสูงสุดที่ภาพจะถูก resize ก่อนเข้าโมดูลตรวจจับ

พารามิเตอร์เหล่านี้ปรับได้ที่ระดับ inference โดยไม่ต้อง fine-tune หรือฝึกโมเดลใหม่ ในโครงงานนี้พบว่าการลดค่า **box_thresh** ช่วยกู้คืนกรอบข้อความจริงที่โมดูลตรวจจับ "เห็น" อยู่แล้วแต่ถูกทิ้งไปเพราะคะแนนความเชื่อมั่นไม่ถึงเกณฑ์เดิม เป็นการเพิ่ม Recall ล้วน ๆ โดยไม่ต้องมีข้อมูลหรือการฝึกโมเดลเพิ่มเติม (รายละเอียดวิธีการและผลการทดลองในหัวข้อ 3.9 และ 4.2)

## 2.5 OpenCV

OpenCV (Open Source Computer Vision Library) เป็นไลบรารีสำหรับการประมวลผลภาพและคอมพิวเตอร์วิทัศน์ที่ได้รับความนิยมอย่างแพร่หลาย ในโครงงานนี้ OpenCV ถูกใช้สำหรับการอ่าน/เขียนไฟล์ภาพ (`cv2.imread`, `cv2.imwrite`) และการเพิ่มขอบภาพขาวก่อนเข้าสู่ OCR (`cv2.copyMakeBorder`)

## 2.6 Data Augmentation ด้วย Albumentations

Data Augmentation คือกระบวนการสร้างข้อมูลเพิ่มเติมจากข้อมูลต้นฉบับ โดยปรับแต่งคุณลักษณะบางอย่างของข้อมูลเพื่อจำลองสถานการณ์การใช้งานจริงที่หลากหลายขึ้น ในงานด้าน Computer Vision มักใช้เพื่อจำลองสภาพแวดล้อมของภาพที่แตกต่างกัน เช่น แสง มุมกล้อง หรือคุณภาพของภาพ

**Albumentations** เป็นไลบรารีสำหรับทำ Data Augmentation บนภาพที่ได้รับความนิยมในงานด้าน Computer Vision เนื่องจากมีความเร็วสูงและรองรับ transform หลากหลายรูปแบบ ในโครงงานนี้ใช้ Albumentations สร้างภาพจำลองสภาพแวดล้อม 4 รูปแบบจากภาพต้นฉบับ ออกแบบให้จำลองลักษณะการถ่ายภาพจริงของผู้ใช้ทั่วไปอย่างละเอียด (มือสั่น, กล้องปรับโฟกัสอัตโนมัติ, การบีบอัด JPEG ตอนส่งรูป, ISO noise จากกล้องมือถือ) มากกว่าการปรับความสว่าง/คอนทราสต์อย่างง่ายเพียงอย่างเดียว (รายละเอียดพารามิเตอร์ในหัวข้อ 3.4)

## 2.7 Fuzzy String Matching

Fuzzy String Matching เป็นเทคนิคที่ใช้เปรียบเทียบความคล้ายคลึงระหว่างข้อความสองชุด โดยสามารถรองรับกรณีที่ข้อความมีข้อผิดพลาดหรือมีการสะกดที่ไม่ตรงกันทั้งหมด เทคนิคดังกล่าวได้รับความนิยมในงานด้านการค้นหาข้อมูล การตรวจสอบการสะกดคำ และการจับคู่ข้อความที่ไม่สมบูรณ์

ในโครงงานนี้ Fuzzy String Matching ถูกนำมาใช้เพื่อเปรียบเทียบข้อความที่ได้จาก OCR กับฐานข้อมูลรายชื่อสารเครื่องสำอาง (INCI) เนื่องจากข้อความที่ได้จาก OCR อาจมีข้อผิดพลาดจากคุณภาพของภาพหรือสภาพแวดล้อมในการถ่ายภาพ

## 2.8 RapidFuzz Library

RapidFuzz เป็นไลบรารีสำหรับการเปรียบเทียบข้อความที่ได้รับการพัฒนาขึ้นเพื่อให้มีประสิทธิภาพสูงและสามารถทำงานได้รวดเร็วกว่าไลบรารี FuzzyWuzzy ในหลายกรณี รองรับอัลกอริทึมการวัดความคล้ายคลึงของข้อความหลายรูปแบบ เช่น `ratio`, `partial_ratio`, `token_sort_ratio`, `token_set_ratio`

ในโครงงานนี้ กระบวนการ Fuzzy Matching หลัง OCR (post-processing) ทำงานดังนี้
1. รวมข้อความที่ OCR อ่านได้ทั้งหมด ตัด header เช่น "Ingredients:" ออกด้วย regular expression
2. แบ่งข้อความเป็น token ที่ comma หรือ colon (ไม่ตัดกรณี comma ตามด้วยตัวเลข เพื่อกันไม่ให้ตัดกลางชื่อสารที่มีตัวเลข เช่น "1,2-Hexanediol")
3. แต่ละ token ที่ยาวตั้งแต่ 3 ตัวอักษรขึ้นไป จะถูกค้นหาชื่อที่ใกล้เคียงที่สุดในฐานข้อมูล INCI (11,290 ชื่อไม่ซ้ำ รวมคำศัพท์เสริม — ดูหัวข้อ 3.7) ด้วยฟังก์ชัน `rapidfuzz.process.extractOne` โดยใช้ scorer แบบ `fuzz.ratio` และกำหนด **score_cutoff = 90**
4. รับผลการจับคู่ก็ต่อเมื่อผ่านเงื่อนไขเพิ่มเติมเรื่องอัตราส่วนความยาวตัวอักษร (ระหว่าง token กับชื่อที่จับคู่ได้) ไม่เกิน 1.6 เท่า เพื่อกันการจับคู่ผิดระหว่าง token สั้นกับชื่อสารที่ยาวกว่ามาก
5. หากไม่ผ่านเงื่อนไข จะเก็บ token เดิมไว้โดยไม่แก้ไข

### 2.8.1 หลักการทำงานของ Levenshtein Distance

Levenshtein Distance เป็นวิธีการวัดความแตกต่างระหว่างข้อความสองชุด โดยคำนวณจากจำนวนครั้งขั้นต่ำของการดำเนินการที่จำเป็นในการแปลงข้อความหนึ่งให้เป็นอีกข้อความหนึ่ง ซึ่งประกอบด้วยการเพิ่มอักขระ (Insertion) การลบอักขระ (Deletion) และการแทนที่อักขระ (Substitution)

เทคนิคดังกล่าวเป็นพื้นฐานสำคัญของไลบรารี RapidFuzz ในการคำนวณค่าความคล้ายคลึงของข้อความสำหรับขั้นตอน Fuzzy Matching และยังเป็นพื้นฐานของตัวชี้วัด **Character Error Rate (CER)** และ **Word Error Rate (WER)** ที่ใช้ในการประเมินผลของโครงงานนี้ (รายละเอียดสูตรคำนวณอยู่ในบทที่ 3)

## 2.9 งานวิจัยที่เกี่ยวข้อง

Baek และคณะ (2019) ได้ศึกษาและเปรียบเทียบโมเดล Scene Text Recognition หลายรูปแบบ โดยพบว่าคุณภาพของภาพมีผลโดยตรงต่อประสิทธิภาพของ OCR โดยเฉพาะในกรณีของข้อความขนาดเล็กและพื้นหลังที่ซับซ้อน ซึ่งสอดคล้องกับข้อค้นพบในโครงงานนี้ที่ภาพความละเอียดต่ำส่งผลกระทบต่อความแม่นยำของ OCR อย่างชัดเจน และเป็นคอขวดหลักที่เหลืออยู่ (ดูหัวข้อ 4.3)

Du และคณะ (2020) ได้นำเสนอ PP-OCR ซึ่งเป็นระบบ OCR ที่มีขนาดเล็กและประสิทธิภาพสูง เป็นต้นตระกูลของโมเดล PP-OCRv6 ที่ใช้เป็น OCR Engine หลักในงานวิจัยนี้ รวมถึงใช้สถาปัตยกรรม DB (Differentiable Binarization) สำหรับการตรวจจับข้อความ ซึ่งเป็นส่วนที่โครงงานนี้ปรับ threshold เพื่อเพิ่มความแม่นยำ (หัวข้อ 2.4)

V. I. Levenshtein (1966) ได้นำเสนอแนวคิด Levenshtein Distance ซึ่งเป็นพื้นฐานสำคัญของไลบรารี RapidFuzz และตัวชี้วัด CER/WER ที่ใช้ในโครงงานนี้

Buslaev และคณะ (2020) ได้นำเสนอ **Albumentations** ไลบรารีสำหรับทำ Data Augmentation บนภาพที่มีความเร็วสูงและรองรับ transform หลากหลายรูปแบบ ซึ่งใช้ในการสร้างชุดข้อมูลจำลองสภาพแวดล้อม 4 รูปแบบของโครงงานนี้

จากการศึกษางานวิจัยที่เกี่ยวข้องพบว่า แม้จะมีงานวิจัยจำนวนมากที่มุ่งเน้นการพัฒนาเทคนิค OCR และเทคนิคปรับปรุงคุณภาพภาพแยกกัน แต่ยังมีงานวิจัยจำนวนน้อยที่กล่าวถึงผลของการตั้งค่า (configuration) โมดูล OCR สำเร็จรูปอย่างละเอียด ทั้งที่ในทางปฏิบัติพบว่าการตั้งค่าที่ไม่เหมาะสมส่งผลกระทบต่อความแม่นยำมากกว่าการเพิ่มเทคนิค preprocessing เชิงภาพเสียอีก โครงงานนี้จึงมุ่งเน้นกระบวนการวินิจฉัยและแก้ไขการตั้งค่า pipeline อย่างเป็นระบบ ก่อนนำไปพัฒนาต่อและใช้งานจริงในระบบแนะนำสกินแคร์ที่เหมาะสมกับผู้ใช้

# บทที่ 3 วิธีดำเนินงาน (ฉบับปรับปรุงครั้งที่ 2)

## 3.1 ภาพรวมของระบบ

งานวิจัยนี้ดำเนินการในลักษณะการวินิจฉัยและปรับแต่ง pipeline OCR เชิงประจักษ์แบบวนซ้ำ (iterative empirical pipeline diagnosis and repair) โดยใช้ PaddleOCR เป็นเครื่องมือรู้จำข้อความหลัก บนภาพฉลากส่วนผสมเครื่องสำอางชุดเดียวกัน เพื่อค้นหาการตั้งค่าที่ให้ความแม่นยำสูงที่สุดสำหรับนำไปพัฒนาต่อและผนวกเข้ากับระบบแนะนำสกินแคร์ที่เหมาะสมกับผู้ใช้ กระบวนการดำเนินงานแบ่งเป็น 4 ขั้นตอนหลัก ได้แก่ (1) การเตรียมชุดข้อมูลและจำลองสภาพแวดล้อม (2) การวินิจฉัยและแก้ไขการตั้งค่า engine หลัก (3) การเสริมฐานข้อมูลคำศัพท์และปรับ threshold การตรวจจับข้อความ และ (4) การประเมินผลด้วยตัวชี้วัดระดับชื่อสารและ CER/WER

## 3.2 สถาปัตยกรรมของระบบ

ระบบที่พัฒนาขึ้นประกอบด้วยขั้นตอนหลัก ดังนี้

1. การรับภาพฉลากเครื่องสำอาง (Input Image) — ภาพจริงหรือภาพจำลองสภาพแวดล้อม
2. เพิ่มขอบภาพขาว 25% รอบภาพ (Border Padding — หัวข้อ 3.6)
3. การดึงข้อความด้วย OCR (PaddleOCR PP-OCRv6_medium, CPU, document unwarping + textline orientation เปิดใช้งาน, text detection box_thresh = 0.30 — หัวข้อ 3.5)
4. การจับคู่ข้อความด้วย Fuzzy Matching (RapidFuzz) เทียบกับฐานข้อมูล INCI (หัวข้อ 3.7-3.8)
5. การคำนวณตัวชี้วัดระดับชื่อสาร (Set Precision/Recall/F1, IER) และ CER/WER เทียบกับ Ground Truth
6. การบันทึกผลลัพธ์รายภาพและสรุปผลเฉลี่ยแยกตามสภาพแวดล้อม

แผนภาพสถาปัตยกรรม (concept): `Input Image → Border Padding (25%) → PaddleOCR (unwarping + textline orientation, box_thresh=0.30) → Fuzzy Matching (RapidFuzz, threshold=90) → CER/WER + Set-F1/IER Evaluation → results/<run>.csv`

## 3.3 การสร้างชุดข้อมูล

### 3.3.1 ภาพต้นฉบับและ Ground Truth

ชุดข้อมูลภาพต้นฉบับ (`test_SkinSafe/image/*.jpg`) เป็นภาพถ่ายฉลากส่วนผสมเครื่องสำอางจริงจำนวน **216 ภาพ** ที่ถูก crop มาเฉพาะส่วนที่มีรายชื่อสารเคมี ขนาดภาพมีความหลากหลายและเล็กกว่าภาพถ่ายฉลากทั่วไปมาก โดยมีความสูงต่ำสุด 70 พิกเซล สูงสุด 1,000 พิกเซล มัธยฐาน (median) 170.5 พิกเซล และความกว้างมัธยฐาน 532 พิกเซล (อัตราส่วนกว้าง:สูง มัธยฐาน ≈ 3.08:1)

Ground Truth (`test_SkinSafe/label/label<ชื่อไฟล์>.txt`) มีครบทั้งหมด **216 ไฟล์** (จับคู่ครบทุกภาพ) แต่ละไฟล์คือรายชื่อสาร (INCI name) หนึ่งชื่อต่อบรรทัด ได้จากการถอดข้อความด้วยมือ/ผู้ช่วย AI จากภาพฉลากจริง

Reference text ที่ใช้เทียบกับผล OCR คือการนำรายชื่อสารทุกชื่อในไฟล์ label มาต่อกันด้วย `, ` (comma + space) เป็น string เดียว

## 3.4 Data Augmentation

เพื่อจำลองสภาพแวดล้อมการถ่ายภาพจริงที่มีคุณภาพต่ำ ใช้ไลบรารี **albumentations** สร้างภาพจำลอง 4 สภาพแวดล้อมจากภาพต้นฉบับทุกภาพ (216 × 4 = 864 ภาพ) รวมกับภาพต้นฉบับเป็น **5 สภาพแวดล้อมทั้งหมด รวม 1,080 ภาพ**

| Scenario | จำลองสถานการณ์ | Transform หลัก |
|---|---|---|
| **blur** | มือสั่น/โฟกัสไม่ทันตอนถ่ายฉลากใกล้ ๆ ด้วยมือถือ | `MotionBlur`/`GaussianBlur`/`Defocus` (สุ่มเลือก) + `Downscale` (รายละเอียดหายจากการปรับโฟกัสเร็ว) + `ImageCompression` (บีบอัด JPEG ตอนส่งรูป) |
| **low_light** | ถ่ายในที่แสงน้อย (ร้าน/บ้านไฟสลัว) ยังไม่มืดสนิท | `RandomBrightnessContrast` (ลด) + `RandomGamma` + `ISONoise` (กล้องดัน ISO สู้แสง) + `ImageCompression` |
| **night** | ถ่ายกลางคืน แสงน้อยมาก มักมีแหล่งแสงจุดเดียว มือสั่นเพราะสปีดชัตเตอร์ต่ำ | `RandomBrightnessContrast` (ลดมากกว่า low_light) + `RandomGamma` + `ISONoise` (สูง) + `MotionBlur` + `RGBShift` (แสงไฟโซเดียม/ไฟถนน) + `ImageCompression` |
| **indoor** | ถ่ายในร่ม แสงไฟหลอด (เหลือง/ขาว) แสงไม่สม่ำเสมอ อาจมีเงาบัง | `RGBShift` (แสงหลอดไส้/วอร์มไวท์) + `RandomBrightnessContrast` (เล็กน้อย) + `RandomShadow` + `ImageCompression` |

แต่ละภาพสุ่มค่าภายในช่วง limit ที่กำหนดเพียงครั้งเดียวตอนสร้าง (`augmented_dataset/` ถูก generate ไว้ล่วงหน้าด้วยสคริปต์ `augument.py` ไม่สุ่มใหม่ทุกครั้งที่รัน OCR) ทำให้ทุกการทดลองเห็นภาพชุดเดียวกันเป๊ะ ๆ เปรียบเทียบกันได้แบบ apples-to-apples

## 3.5 เครื่องมือ OCR และการตั้งค่า

ใช้ **PaddleOCR 3.7.0** โมเดล **PP-OCRv6_medium** รันบน **CPU** การตั้งค่าที่ใช้จริงและเหตุผลของแต่ละค่า:

```python
PaddleOCR(
    lang="en",
    use_doc_orientation_classify=False,
    use_doc_unwarping=True,       # เปิด: ลด mean CER เทียบกับปิด (validated)
    use_textline_orientation=True, # เปิด: ลด mean CER เทียบกับปิด (validated)
    device="cpu",                  # ดูเหตุผลด้าน hardware ในหัวข้อ 3.11
    enable_mkldnn=False,           # ปิด: หลีกเลี่ยงบัค crash ("ConvertPirAttribute2RuntimeAttribute not support ...")
    text_det_box_thresh=0.30,      # ลดจาก default 0.45 — เพิ่ม Recall การตรวจจับข้อความ (หัวข้อ 3.9)
)
```

โมเดล PaddleOCR ถูก cache เป็น singleton และใช้ซ้ำข้ามทุกภาพในรอบทดลองเดียวกัน เพราะการโหลดโมเดลใช้เวลานาน เวลาที่วัด (`ocr_time_sec`) จึงเป็นเวลา inference ต่อภาพเท่านั้น (เฉลี่ย ~16 วินาที/ภาพบน CPU ของเครื่องที่ใช้พัฒนา)

## 3.6 Border Padding

ก่อนส่งภาพเข้าสู่ `ocr.predict()` จะเพิ่มขอบภาพสีขาวรอบภาพด้วย `cv2.copyMakeBorder` โดยขนาดขอบคำนวณจาก `pad_px = round(min(height, width) × 0.25)` (25% ของด้านที่สั้นกว่า) ค่านี้ผ่านการทดลองเปรียบเทียบ (n=40, "original") พบว่า 25% ให้ผลดีที่สุด เพิ่ม Set F1 จาก 0.548 เป็น 0.563 ส่วนใหญ่มาจากการเพิ่ม Recall (0.535→0.581) การ pad มากกว่านี้ (50%) หรือน้อยกว่านี้ (10%) ให้ผลแย่กว่า

## 3.7 ฐานข้อมูลส่วนผสมเครื่องสำอาง (INCI)

ฐานข้อมูลส่วนผสมหลัก (`ingredient_master_dataset_fixed.csv`) มีรายชื่อสาร INCI จำนวน **11,091 รายการไม่ซ้ำ** จากการตรวจสอบพบว่าฐานข้อมูลนี้ขาดชื่อสาร INCI ที่พบบ่อยมากในฉลากจริงไปประมาณ 58.6% ของชื่อสารที่ปรากฏจริง (เช่น Glycerin, Niacinamide, Fragrance, Aqua) จึงเสริมด้วย **ไฟล์ patch คำศัพท์ที่คัดสรรแล้ว 199 รายการ** (`common_ingredients_patch.csv`) คัดจากชื่อสารที่ปรากฏในฉลากจริงตั้งแต่ 5 ผลิตภัณฑ์ขึ้นไปแต่ไม่มีอยู่ในฐานข้อมูลหลัก รวมเป็น **11,290 รายการไม่ซ้ำ**

ข้อสังเกตสำคัญจากการทดลอง: การเสริมคำศัพท์แบบคัดสรรเฉพาะชื่อที่พบบ่อยเช่นนี้ช่วยเพิ่มความแม่นยำอย่างวัดผลได้ ในขณะที่การทดลองเสริมคำศัพท์แบบ "ใส่ทุกชื่อที่หาได้" (2,023 ชื่อ) กลับทำให้ผลลัพธ์แย่ลง เนื่องจากมีตัวเลือกที่คล้ายกันมากเกินไปทำให้ fuzzy matching จับคู่ผิดง่ายขึ้น

## 3.8 Fuzzy String Matching (การนำไปใช้จริง)

รายละเอียดขั้นตอนตรงกับหัวข้อทฤษฎี 2.8 โดยพารามิเตอร์ที่ใช้จริงคือ score_cutoff = 90 (สูงกว่าค่าตั้งต้น 85 ที่เคยทดลองในช่วงแรก — sweep พบว่า 90 ให้ mean CER ต่ำกว่า 85 โดยไม่มีข้อเสีย) และแบ่ง token ด้วย comma หรือ colon

## 3.9 การปรับ Detection Box Threshold

เนื่องจากพบว่าคอขวดสำคัญที่เหลืออยู่หลังแก้ไข pipeline หลักคือค่า Recall ของการตรวจจับข้อความ จึงทดลองปรับพารามิเตอร์ post-processing ของโมดูลตรวจจับข้อความ (หัวข้อ 2.4) โดยไม่ fine-tune โมเดล วิธีดำเนินการ:

1. **Sweep เบื้องต้น** บนตัวอย่าง 40 ภาพ ("original") เปรียบเทียบ 5 ค่า config: baseline (ค่าเริ่มต้น), ลด `box_thresh` เป็น 0.30, เพิ่ม `unclip_ratio` เป็น 1.8, ผสมทั้งสอง, และเพิ่ม `limit_side_len` เป็น 1536
2. **Sweep ละเอียดเพิ่มเติม** ทดสอบ `box_thresh` ที่ 0.30, 0.25, 0.20, 0.15 เพื่อหาว่ายังมีกำไรเพิ่มหรือไม่หากลดต่ำกว่า 0.30
3. **ตรวจสอบความน่าเชื่อถือ (sanity check)**: ยืนยันว่าผลต่างที่พบไม่ได้มาจากความไม่คงที่ของโมเดล (non-determinism) โดยรันค่า config เดิมซ้ำหลายรอบบนภาพชุดเดียวกัน พบว่าผลลัพธ์เหมือนเดิมทุกครั้ง (deterministic) จึงสรุปได้ว่าผลต่างที่วัดได้เป็นผลจากพารามิเตอร์จริง
4. **Validate เต็มรูปแบบ** บนภาพทั้งหมด 216 ภาพด้วยค่าที่ดีที่สุดจากการ sweep

ผลการทดลองแสดงในหัวข้อ 4.2

## 3.10 การประเมินผล

### 3.10.1 การ Normalize ข้อความก่อนเทียบ

```python
def normalize_text(text):
    text = text.lower().strip()
    text = re.sub(r"\s+", " ", text)
    return text
```

ทำกับทั้ง OCR output (hypothesis) และ reference text ก่อนคำนวณเสมอ เพื่อไม่ให้ case-sensitivity หรือช่องว่างเกินกระทบคะแนน

### 3.10.2 Character Error Rate (CER) และ Word Error Rate (WER)

```
CER = Levenshtein_distance(normalize(hypothesis), normalize(reference)) / len(normalize(reference))
WER = Levenshtein_distance(normalize(hypothesis).split(), normalize(reference).split()) / len(normalize(reference).split())
```

คำนวณด้วย `rapidfuzz.distance.Levenshtein.distance` ค่ายิ่งน้อยยิ่งดี ทั้งสองค่าคำนวณแยกต่อภาพแล้วหาค่าเฉลี่ยเลขคณิต (macro-average)

### 3.10.3 ตัวชี้วัดระดับชื่อสาร (Ingredient-level Metrics) — ตัวชี้วัดหลัก

เนื่องจาก CER/WER เปรียบเทียบที่ระดับตัวอักษร/คำ ซึ่งได้รับผลกระทบจากความแตกต่างของรูปแบบการจัดวางข้อความ (reference ต่อด้วย comma ในขณะที่ OCR อ่านตามการตัดบรรทัดจริงบนฉลาก) โครงงานนี้จึงใช้ตัวชี้วัดเสริมที่เปรียบเทียบที่ **ระดับชื่อสาร (ingredient token)** เป็นตัวชี้วัดหลักในการตัดสินใจปรับแต่ง pipeline (`compute_ingredient_metrics.py`):

1. ตัดข้อความทั้งสองฝั่ง (OCR output และ reference) เป็น token ด้วย regex ที่ครอบคลุม comma, colon และ period (period มักเป็น comma ที่ OCR อ่านผิด — parse เฉพาะตอนให้คะแนน ไม่ใช่ตอนแก้ไขข้อความจริง เพราะทดสอบแล้วพบว่าการ split ที่ period ในขั้นตอนแก้ไขข้อความทำให้ fragment เล็กลงและจับคู่ผิดง่ายขึ้น)
2. **Ingredient Error Rate (IER)** = Levenshtein distance ระหว่าง token list สองฝั่ง หารด้วยจำนวน token ของ reference — นับ token ทั้งก้อนถูก/ผิด ไม่มี partial credit
3. **Set Precision/Recall/F1** = จับคู่ token ของทั้งสองฝั่งแบบ one-to-one ด้วย `rapidfuzz.fuzz.ratio` (เรียงจากคะแนนสูงไปต่ำ, threshold ≥ 85 คะแนน) แทนการเทียบแบบ exact match เพื่อให้เครดิตกับคำที่ใกล้เคียงถูกต้อง — **ใช้ Set F1 เป็นตัวชี้วัดหลักของโครงงานนี้** เพราะตรงกับเป้าหมายเชิงปฏิบัติ (ระบุชื่อสารได้ถูกต้องหรือไม่) มากกว่า CER/WER หรือ IER แบบเข้มงวด

โมเดล OCR รันบน **CPU เท่านั้น** แม้เครื่องพัฒนามี GPU (NVIDIA RTX 5060) เพราะทดสอบพบว่า `paddlepaddle-gpu` บน GPU รุ่นนี้ (compute capability 12.0, สถาปัตยกรรม Blackwell) ให้ผลลัพธ์ผิดพลาดแบบเงียบ (silent garbage output ไม่มี error ใด ๆ) เนื่องจาก compiled kernel ของเวอร์ชันที่ใช้ได้ยังไม่รองรับ compute capability รุ่นนี้ การใช้ CPU inference จึงเป็นทางเลือกเดียวที่ให้ผลลัพธ์ถูกต้องบนเครื่องนี้ในขณะทำวิจัย

# บทที่ 4 ผลการทดลองและการวิเคราะห์ (ฉบับปรับปรุงครั้งที่ 2)

## 4.1 ผลกระทบของการแก้ไข Pipeline หลัก

ก่อนการแก้ไข พบว่าค่า Set F1 บนภาพจริง 216 ภาพ ("original") อยู่ที่ **0.39** หลังวินิจฉัยและแก้ไขปัญหา 4 จุด ได้แก่ (1) บัค mkldnn ที่ทำให้ crash และต้องปิดการใช้งาน (2) การตั้งค่า engine ผิด (`use_doc_unwarping`/`use_textline_orientation` เคยถูกปิดไว้) (3) ช่องโหว่ฐานข้อมูลคำศัพท์ INCI ที่ขาดชื่อสารที่พบบ่อย และ (4) บัคในขั้นตอน post-processing ของการแก้ไขข้อความ ค่า Set F1 ดีขึ้นเป็น **0.55 (+41%)**

## 4.2 ผลการปรับ Detection Box Threshold

### 4.2.1 ผล Sweep เบื้องต้น (n=40, "original")

| Config | Precision | Recall | Set F1 |
|---|---|---|---|
| Baseline (box_thresh=0.45) | 0.579 | 0.581 | 0.563 |
| **box_thresh=0.30** | 0.565 | **0.632** | **0.592** |
| box_thresh=0.30 + unclip_ratio=1.8 | 0.514 | 0.560 | 0.530 |
| unclip_ratio=1.8 (เดี่ยว) | 0.525 | 0.528 | 0.512 |
| limit_side_len=1536 | 0.425 | 0.426 | 0.409 |

การลด `box_thresh` เพียงอย่างเดียวให้ผลดีที่สุด สอดคล้องกับสมมติฐานว่าคอขวดคือ Recall ของการตรวจจับ ส่วนการเพิ่ม `unclip_ratio` หรือ `limit_side_len` กลับให้ผลแย่กว่า baseline ทั้งคู่

### 4.2.2 การหาจุด Plateau ของ box_thresh

ทดสอบ `box_thresh` เพิ่มเติมที่ 0.25, 0.20, 0.15 บนตัวอย่างชุดเดียวกัน (n=40) พบว่าทั้งสามค่าให้ผลลัพธ์ **เหมือนกับ box_thresh=0.30 ทุกประการ** (ตรวจสอบระดับข้อความ OCR ราย, พบว่า raw text เหมือนกันทุกตัวอักษร) แสดงว่ากรอบข้อความที่กู้คืนได้ด้วยการลด threshold ถูกกู้คืนหมดแล้วที่ 0.30 — ข้อความที่ยังพลาดอยู่หลังจากนี้คือกรณีที่โมดูลตรวจจับ "มองไม่เห็น" จริง ๆ ไม่ใช่แค่ถูกกรองทิ้งด้วย threshold **จึงสรุปว่า 0.30 คือค่าที่เหมาะสมที่สุดสำหรับพารามิเตอร์นี้ ไม่มีประโยชน์เพิ่มเติมจากการลดต่ำกว่านี้**

### 4.2.3 ผลการ Validate เต็มรูปแบบ (n=216, "original")

| ตัวชี้วัด | box_thresh=0.45 (ก่อนปรับ) | box_thresh=0.30 (ปัจจุบัน) |
|---|---|---|
| mean CER (fuzzy) | 0.3199 | **0.2644** |
| mean IER | 0.6709 | 0.6757 |
| mean Set Precision | 0.5604 | 0.5487 |
| mean Set Recall | 0.5780 | **0.6127** |
| **mean Set F1** | 0.5515 | **0.5715** |

ผลยืนยันบนภาพเต็มสอดคล้องกับ sweep ตัวอย่างย่อย: Set F1 ดีขึ้นจาก 0.5515 เป็น 0.5715 (+3.6%) และ CER ดีขึ้นชัดเจนกว่า (-17%) โดยแลกกับ Precision ที่ลดลงเล็กน้อย เป็นการปรับปรุงที่ไม่ต้องใช้ข้อมูลหรือการฝึกโมเดลเพิ่มเติมเลย ค่า `text_det_box_thresh=0.30` นี้ถูกกำหนดเป็นค่า default ถาวรใน `ocr_core.py`

## 4.3 คอขวดที่เหลืออยู่: Recall จากการตรวจจับข้อความ

จากการวิเคราะห์เชิงคุณภาพพบว่าประมาณ **62.7%** ของชื่อสารที่ระบบพลาดไม่ใช่เพราะแก้ไขข้อความผิดพลาด แต่เป็นเพราะโมดูลตรวจจับข้อความ (text detector) ไม่พบกรอบข้อความบริเวณนั้นเลยตั้งแต่ต้น ซึ่งเป็นข้อจำกัดที่การปรับพารามิเตอร์ระดับ inference (หัวข้อ 3.9, 4.2) แก้ไขได้เพียงบางส่วนเท่านั้น (ดูข้อสรุปหัวข้อ 4.2.2) การแก้ปัญหานี้ต่อไปน่าจะต้องอาศัยการปรับปรุงคุณภาพ/ความละเอียดของภาพต้นทาง หรือการ fine-tune โมเดลตรวจจับข้อความให้เหมาะกับลักษณะฉลากเครื่องสำอางโดยเฉพาะ ซึ่งต้องใช้ข้อมูล bounding-box annotation เพิ่มเติมและทรัพยากร GPU สำหรับการฝึกโมเดล (เกินขอบเขตของโครงงานระยะนี้)

## 4.4 การพิจารณาโมเดล OCR ทางเลือก

นอกจาก PaddleOCR ยังได้ทดลองโมเดล OCR/Vision-Language แบบ generative อีก 2 ตัวเพื่อเปรียบเทียบเชิงคุณภาพ:

- **GOT-OCR2.0** (`stepfun-ai/GOT-OCR-2.0-hf`) — พบอาการ hallucinate รุนแรงบนภาพคุณภาพต่ำ (แต่งข้อความที่ฟังดูสมเหตุสมผลแต่ไม่เกี่ยวกับภาพเลย) ในสัดส่วนสูงราว 32% ของภาพทดสอบ ถือเป็น catastrophic failure ที่ไม่พบใน PaddleOCR
- **Qwen2.5-VL-3B (4-bit quantized)** — ให้ผลแย่กว่า PaddleOCR อย่างชัดเจน และมีอาการแต่งชื่อสารเคมีปลอมที่ฟังดูน่าเชื่อถือ (hallucinate เฉพาะทางเคมี) ซึ่งเป็นความเสี่ยงสูงสำหรับระบบที่เกี่ยวข้องกับความปลอดภัยของผู้บริโภค

ผลการเปรียบเทียบเชิงคุณภาพนี้สนับสนุนการเลือกใช้ PaddleOCR (แบบ detection+recognition เดิม) ต่อไป แทนที่จะเปลี่ยนไปใช้โมเดล generative/VLM สำหรับ use case นี้

## 4.5 งานที่ยังไม่เสร็จสมบูรณ์ / แผนต่อไป

1. **การ validate เต็มรูปแบบของ 4 สภาพแวดล้อมเสริม** (blur, indoor, low_light, night) ด้วยค่า config ล่าสุด (รวม `text_det_box_thresh=0.30`) ยังไม่เสร็จสมบูรณ์ — ปัจจุบันมีเฉพาะผลการ validate เต็มรูปแบบของ "original" (216 ภาพ) เท่านั้น ผลของ 4 สภาพแวดล้อมเสริมจะดำเนินการและรายงานเพิ่มเติมในลำดับถัดไป
2. **เป้าหมาย Set F1 ที่ 0.70** ประเมินว่าไม่สามารถทำได้ด้วยการปรับพารามิเตอร์ (config tuning) เพียงอย่างเดียวอีกต่อไป เนื่องจากคอขวด Recall จากการตรวจจับข้อความ (หัวข้อ 4.3) ต้องอาศัยการปรับปรุงคุณภาพภาพต้นทางหรือการ fine-tune โมเดลตรวจจับข้อความเฉพาะทาง

## 4.6 การอภิปรายผล

ผลการทดลองแสดงให้เห็นว่า**การวินิจฉัยและแก้ไขการตั้งค่า (configuration) ของโมดูล OCR สำเร็จรูปอย่างเป็นระบบ ให้ผลกระทบต่อความแม่นยำมากกว่าการเพิ่มเทคนิค image preprocessing เชิงคลาสสิกอย่างมีนัยสำคัญ** การแก้ไข engine settings ที่ตั้งผิด (เปิด document unwarping/textline orientation), การเพิ่ม border padding, การเสริมฐานข้อมูลคำศัพท์ และการปรับ detection threshold รวมกันทำให้ Set F1 ขยับจาก 0.39 เป็น 0.57 (+46%) โดยไม่ต้องใช้เทคนิค preprocessing เชิงภาพแบบดั้งเดิมแม้แต่เทคนิคเดียว

ประเด็นสำคัญที่ควรเน้นในการอภิปราย:

1. **การตั้งค่าฟีเจอร์ในตัวของโมเดลผิดพลาดสร้างความเสียหายมากกว่าที่คาด** — การปิด document unwarping/textline orientation (ซึ่งเดิมตั้งใจปิดเพื่อทดสอบ preprocessing แบบแยกส่วนอย่างเป็นธรรม) กลับเป็นการเสียโอกาสความแม่นยำที่โมเดลมีอยู่แล้วโดยไม่จำเป็น
2. **การปรับพารามิเตอร์ inference ของโมดูลตรวจจับข้อความ (box_thresh) ให้ผลตอบแทนสูงเทียบกับต้นทุน** เพราะไม่ต้องมีข้อมูลหรือฝึกโมเดลเพิ่มเติมเลย แต่มีจุด plateau ชัดเจน (0.30) ที่หลังจากนั้นต้องอาศัยการ fine-tune จริงจึงจะไปต่อได้
3. **การเสริมฐานข้อมูลคำศัพท์ต้องคัดสรรอย่างระมัดระวัง** — ยิ่งใส่คำศัพท์มากไม่ได้แปลว่ายิ่งดีเสมอไป (การใส่แบบ "oracle" ทุกชื่อกลับทำให้แย่ลง) ควรคัดเฉพาะคำที่พบบ่อยจริงในข้อมูลเป้าหมาย
4. **คอขวดที่เหลืออยู่ (Recall จากการตรวจจับข้อความ) เป็นข้อจำกัดเชิงโครงสร้างของโมเดลสำเร็จรูป** ที่การปรับพารามิเตอร์ระดับ inference แก้ได้เพียงบางส่วน สะท้อนว่าขั้นต่อไปของงานวิจัยนี้ควรมุ่งไปที่คุณภาพภาพต้นทางหรือการ fine-tune โมเดลเฉพาะทาง มากกว่าการค้นหาพารามิเตอร์เพิ่มเติม

### ข้อสรุปภาพรวม

สำหรับชุดข้อมูลภาพฉลากส่วนผสมที่ crop มาเป็นแถบเล็ก ความละเอียดต่ำ **การวินิจฉัยและแก้ไขการตั้งค่า pipeline ของโมดูล OCR สำเร็จรูปอย่างเป็นระบบ (engine configuration, border padding, ฐานข้อมูลคำศัพท์, detection threshold) ให้ผลตอบแทนสูงกว่าการเพิ่มเทคนิค preprocessing เชิงภาพแบบดั้งเดิมอย่างชัดเจน** ผลลัพธ์นี้ชี้ให้เห็นว่าก่อนลงทุนพัฒนาเทคนิคเสริมที่ซับซ้อน ควรตรวจสอบให้แน่ใจก่อนว่าโมดูลสำเร็จรูปที่ใช้อยู่ถูกตั้งค่าอย่างเหมาะสมแล้วหรือยัง Pipeline ปัจจุบัน (Set F1 = 0.57, CER = 0.26 บนภาพจริง 216 ภาพ) พร้อมนำไปพัฒนาต่อและผนวกเข้ากับระบบแนะนำสกินแคร์ที่เหมาะสมกับผู้ใช้ในขั้นต่อไป โดยงานที่เหลือคือการ validate ครบทุกสภาพแวดล้อมและพิจารณาแนวทาง fine-tuning โมเดลตรวจจับข้อความเพื่อแก้คอขวด Recall ที่เหลืออยู่
